WHAT IS DYNAMIC PROGRAMMING (DP)?
ï‚¡ Definition: DP is a method for solving complex problems by breaking them down into simpler
subproblems and solving each subproblem only once, storing its result.
ï‚¡ Key Idea: Overcome the inefficiency of recursive solutions by avoiding recomputation of overlapping
subproblems.
ï‚¡ Goal: Solve problems efficiently by storing intermediate results (memoization) or building up the solution
iteratively (tabulation).
KEY TERMINOLOGY IN DP
ï‚¡ Subproblem: A smaller, simpler version of the original problem.
ï‚¡ Optimal Substructure: The optimal solution to the problem can be constructed from optimal solutions to its
subproblems.
ï‚¡ Overlapping Subproblems: The same subproblems are solved multiple times in different parts of the recursive
tree.
THE DP APPROACH
ï‚¡ Characterize the structure of an optimal solution.
ï‚¡ Define the value of an optimal solution recursively (recurrence relation).
ï‚¡ Compute the value of an optimal solution (usually in a bottom-up manner).
ï‚¡ Construct the optimal solution (if required).
KEY DP PROPERTIES
ï‚¡ Optimal Substructure: A problem has optimal substructure if its optimal solution can be constructed efficiently
from the optimal solutions of its subproblems.
ï‚¡ Overlapping Subproblems: If the problem involves solving the same subproblems multiple times, DP helps by
storing the results to avoid recomputation.
TOP-DOWN APPROACH (MEMOIZATION)
ï‚¡ Top-Down DP (Memoization): Solve the problem using recursion, but store the results of subproblems in a cache
(often a table) to avoid recalculating them.
ï‚¡ Steps:
ï‚¡ Recursively solve the problem.
ï‚¡ Store the result of each subproblem.
ï‚¡ If the same subproblem is encountered again, use the stored result.
BOTTOM-UP APPROACH (TABULATION)
ï‚¡ Bottom-Up DP (Tabulation): Solve the problem iteratively, starting with the smallest subproblems and building up
to the final solution.
ï‚¡ Steps:
ï‚¡ Start solving the smallest subproblems.
ï‚¡ Use previously solved subproblems to build the solution.
ï‚¡ Tabulate the results in a table.
WHEN TO USE TOP-DOWN VS. BOTTOM-UP
ï‚¡ Top-Down (Memoization):
ï‚¡ Easier to implement and intuitive.
ï‚¡ Suitable when the problem has a complex recursive structure.
ï‚¡ May involve more memory (due to recursion stack).
ï‚¡ Bottom-Up (Tabulation):
ï‚¡ More efficient in terms of time and space.
ï‚¡ Avoids recursion overhead.
ï‚¡ Suitable when the problem has a simple iterative structure.

What is a Greedy
Algorithm?
â€¢ Definition: A greedy algorithm builds
up a solution step by step, always
choosing the option that looks best at
the moment.
â€¢ Key Idea: Make the locally optimal
choice at each step, hoping that these
choices lead to a globally optimal
solution.
â€¢ Examples: Shortest path algorithms,
Huffman encoding, spanning trees,
etc.
Characteristics of
Greedy Algorithms
â€¢ Greedy Choice Property: An optimal
solution can be obtained by choosing a local
optimum at each step.
â€¢ Optimal Substructure: A problem has an
optimal substructure if an optimal solution
to the problem contains optimal solutions to
its subproblems.
â€¢ No Backtracking: Unlike Dynamic
Programming or Backtracking, Greedy
Algorithms do not reconsider choices.
â€¢ Efficient: They typically work in O(n log n) or
O(n) time complexity.
When to Use a Greedy
Algorithm?
â€¢ Use Greedy Algorithms if:
â€¢ The problem exhibits optimal substructure and
greedy choice property.
â€¢ You can sort or prioritize elements based on a
property (e.g., weight, value, or distance).
â€¢ A locally optimal choice leads to a globally
optimal solution.
â€¢ Do NOT use if:
â€¢ The problem requires backtracking or dynamic
programming for optimal results.
â€¢ Future choices depend on previous ones (e.g.,
0/1 Knapsack problem).

Example 2 - Huffman Coding
â€¢ Problem: Given characters and their
frequencies, construct an optimal binary tree
(prefix-free) to minimize encoding length.
â€¢ Steps:
â€¢ Build a priority queue of characters
sorted by frequency.
â€¢ Extract two smallest elements, merge
them, and reinsert into the queue.
â€¢ Repeat until only one tree remains.
Huffman Coding Algorithm (Code)
import heapq
class Node:
 def __init__(self, char, freq):
 self.char = char
 self.freq = freq
 self.left = None
 self.right = None

 def __lt__(self, other):
 return self.freq < other.freq
def huffman_coding(freq_table):
 heap = [Node(char, freq) for char, freq in freq_table.items()]
 heapq.heapify(heap)

 while len(heap) > 1:
 left = heapq.heappop(heap)
 right = heapq.heappop(heap)
 merged = Node(None, left.freq + right.freq)
 merged.left, merged.right = left, right
 heapq.heappush(heap, merged)

 return heap[0] # Root of Huffman Tree
Time Complexity: O(nlogn)O(n \log n)O(nlogn)

Analysis
â€¢ Initialize forest O(n)
â€¢ Initialize heap O(m), m = |E|
â€¢ Loop performed m times
â€¢ In the loop one DeleteMin O(log m)
â€¢ Two Find, each O(log n)
â€¢ One Union (at most) O(1)
â€¢ So worst case O(m log m) = O(m log n)
Time Complexity Summary
â€¢ Recall that m = |E| = O(V2

â€¢ Primâ€™s runs in O((n+m) log n)

â€¢ In practice, Kruskal has a tendency to run faster since graphs

Kruskal's Algorithm (Code)
def kruskal(edges, n):
 edges.sort(key=lambda x: x[2]) # Sort by weight
 parent = list(range(n))

 def find(v):
 if parent[v] != v:
 parent[v] = find(parent[v])
 return parent[v]

 mst = []
 for u, v, w in edges:
 root_u, root_v = find(u), find(v)
 if root_u != root_v:
 mst.append((u, v, w))
 parent[root_u] = root_v # Union operation

 return mst
Time Complexity: O(ElogE)O(E \log E)O(ElogE)

â€¢ Uses a Queue (FIFO)
â€¢ Explores neighbors before moving deeper.
â€¢ Useful for finding shortest path in an unweighted graph.
â€¢ Time Complexity: O(V + E)

1.Push the starting node into the queue.
2.Mark it as visited.
3.While the queue is not empty:
1. Dequeue a node.
2. Process it.
3. Enqueue all unvisited neighbors.
